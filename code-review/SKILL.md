# 代码审核 Skill

## 工作流程

### Step 1：读取代码
读取 `src/` 目录下所有代码文件，快速建立整体结构理解。

### Step 2：读取参考
- `docs/prd.md`：了解功能需求
- `docs/architecture.md`：了解技术选型和接口设计
- `docs/review-report.md`：如已存在，了解上次 Review 的修复情况

### Step 3：逐维度审核

**安全审核（高优先级，任何问题都必须修复）**
- SQL 注入风险（字符串拼接 SQL）
- XSS 漏洞（未转义的用户输入直接渲染）
- 硬编码密钥/密码/Token
- 敏感数据未加密存储
- 接口缺少鉴权保护
- 越权漏洞（用户A能否访问用户B的数据）
- 文件上传缺少类型和大小限制
- 路径穿越风险

**代码质量（中优先级）**
- 函数超过 50 行
- 重复代码
- 无意义命名（data、temp、flag）
- 死代码
- 异常处理不完善（catch 后什么都不做）
- 缺少合理日志记录

**性能审核（中优先级）**
- N+1 查询
- 全表扫描（where 条件字段无索引）
- 循环内重复计算
- 大数据量未分页处理
- 内存泄漏风险

**规范审核（低优先级）**
- 命名不符合规范
- 缺少必要注释
- .env.example 不完整
- README.md 缺少启动说明

### Step 4：输出 Review 报告

```markdown
# Code Review 报告
- 审核时间：YYYY-MM-DD HH:mm
- 总结：发现 X 个问题（高: X / 中: X / 低: X）

## 高优先级问题（必须修复）
### 问题1：[简短标题]
- 文件：src/xxx/xxx.ts 第 XX 行
- 问题描述：...
- 风险：如果不修复，可能导致...
- 修复建议：...

## 中优先级问题（建议修复）
## 低优先级问题（可后续优化）
## 做得好的地方
```

## 约束
- 产出保存到 `docs/review-report.md`
- 不要只说"有问题"，必须给出具体的文件路径、行号、修复建议
- "做得好的地方"至少列出 2 点（正面反馈同样重要）
