# 桑夏写作规范

> 写作不是灵感的迸发，而是可复制的工程。
>
> 本文档将桑夏的写作风格拆解为可执行的步骤和检查项。

---

## 一、核心风格要素

### 1.1 标题公式

**公式**：`"核心概念：解释/问题/副标题"`

**示例**：
```
✅ Skills 是什么？给你的 AI 装上一个"大脑皮层"
✅ Harness 原理：OpenClaw 的技能加载器是如何工作的？
✅ 如何制作符合规范的 Skills？Anthropic 官方最佳实践
✅ 我开源了博客质量保障工具：sangxia-blog-skill

❌ Skills 介绍          # 太简单
❌ 关于 Harness 的研究   # 没有吸引力
```

**变体**：
| 类型 | 公式 | 示例 |
|------|------|------|
| **问答式** | X 是什么？+ 通俗解释 | "Skills 是什么？给你的 AI 装上一个'大脑皮层'" |
| **原理式** | X 原理：+ 具体说明 | "Harness 原理：OpenClaw 的技能加载器是如何工作的？" |
| **教程式** | 如何 X？+ 权威来源 | "如何制作符合规范的 Skills？Anthropic 官方最佳实践" |
| **宣布式** | 我 X 了：+ 具体项目 | "我开源了博客质量保障工具：sangxia-blog-skill" |

---

### 1.2 开场设计

**公式**：`引用块 + 痛点问题 + 价值主张`

**结构**：
```markdown
# 主标题

> 用引用块提出核心问题，引发读者思考
>
> 问题要与读者痛点相关

---

## 从一个问题/场景开始
```

**示例**：
```markdown
# Skills 是什么？给你的 AI 装上一个"大脑皮层"

> 同样的 AI，为什么别人的能自动调用 API、执行工作流，你的却只会聊天？
>
> 问题可能不在模型，在于你有没有给它装上 Skills。

---
```

**变体**：
| 类型 | 开场方式 | 示例 |
|------|---------|------|
| **对比式** | 提出问题 + 暗示答案 | "为什么别人的 AI 能...你的却..." |
| **痛点式** | 直接描述痛点 | "你花了一晚上写了个 skill，结果..." |
| **场景式** | 描述真实场景 | "上周发布...系列文章时，我遇到了..." |

---

### 1.3 场景对比

**目的**：让读者直观感受价值差异

**结构**：
```markdown
### 一个真实场景

**没有 X 的 AI**：

```
用户：指令
AI: 无法完成/需要更多信息
```

**有 X 的 AI**：

```
用户：同样的指令
AI: 完成 + 额外价值
```

**同样的 XX，为什么差异这么大？**
```

**示例**：
```markdown
**没有 Skills 的 AI**：

```
你：帮我查一下北京明天的天气
AI: 我无法访问实时数据，但你可以...
你：那帮我看看 GitHub 上那个 PR 的 CI 过了没
AI: 我需要你提供更多信息，比如...
```

**有 Skills 的 AI**：

```
你：北京明天天气怎么样？
AI: 北京：⛅️ +8°C，湿度 71%，西风 5km/h

你：看看 PR #55 的 CI 过了没
AI: ✅ 全部通过！3 个 workflow 都成功了
```

**同样的 AI 模型，为什么前后差异这么大？**
```

---

### 1.4 类比解释

**原则**：抽象概念用熟悉事物解释

**结构**：
```markdown
### N 个类比帮你理解

| 类比 | 说明 |
|------|------|
| **类比 1** | 说明 1 |
| **类比 2** | 说明 2 |
| **类比 3** | 说明 3 |

**核心逻辑**：
> 用一句话总结
```

**示例**：
```markdown
### 3 个类比帮你理解

| 类比 | 说明 |
|------|------|
| **VS Code 插件** | 编辑器核心很简单，插件让它能写 Python、调试、部署 |
| **手机 App** | 手机是通用设备，App 让它能拍照、导航、点外卖 |
| **员工培训手册** | 新人很聪明，但不懂业务，手册教他具体怎么干活 |

**核心逻辑**：
> AI 模型本身很聪明，但它没有你的**领域知识**和**工作流程**。
>
> Skills 就是把这些东西打包教给它。
```

**类比库**：
| 概念 | 推荐类比 |
|------|---------|
| Skills | VS Code 插件、手机 App、员工培训手册、岗位培训 |
| Harness | 操作系统、浏览器、Android 系统、调度器 |
| API | 餐厅菜单、插座接口、快递柜、服务员 |
| 框架 | 乐高底板、建筑蓝图、模具 |

       │
       ▼
┌─────────────┐


**示例**：
```markdown

```

       ▼

## 二、写作流程

### 2.1 标准流程

```
1. 确定主题和核心观点
   ↓
2. 搜集素材（使用 search-materials.js）
   ↓
3. 搭建文章结构（使用模板）
   ↓
4. 填充内容（场景 + 类比 + 示例）
   ↓
5. 检查风格（使用 check-style.js）
   ↓
6. 发布
```

---

### 2.2 素材搜集

**命令**：
```bash
node scripts/search-materials.js --topic "<主题>" --output materials.md
```

**输出**：
- 本地相关文章
- 网络参考资料
- 写作建议
- 配图建议

---

## 三、内容原则

### 3.1 简洁至上

> 上下文窗口是公共资源（适用于 Skills 写作）
> 读者时间是宝贵资源（适用于所有文章）

**做法**：
- 能用示例说明的，不写解释性文字
- 能用表格展示的，不用纯文本
- 能拆分到 resources 的，不放在正文

**自检问题**：
- "这段文字读者真的需要知道吗？"
- "这个解释是否证明了它的篇幅成本？"
- "能否用示例替代解释？"

---

### 3.2 示例驱动

> 一个示例胜过千言万语

**要求**：
- 所有概念都有示例
- 所有错误都有正确示范
- 所有命令都可执行

**示例格式**：
```markdown
**错误示例**：
```
❌ 错误代码
```

**正确示例**：
```
✅ 正确代码
```
```

---

### 3.3 用户导向

> 读者想要什么，就提供什么

**做法**：
- 预判读者问题（FAQ）
- 提供快速解决方案
- 减少理解成本

## 四、进阶技巧

### 4.1 节奏控制

**段落长度**：
- 短段落（1-3 句）：强调重点
- 中等段落（3-5 句）：常规说明
- 长段落（5+ 句）：深入解释

**分隔线使用**：
```markdown
---
```
每 300-500 字用一个分隔线，帮助读者视觉休息

---

### 4.2 强调技巧

**加粗**：强调关键词
> Skills 就是 **AI 的"岗位培训手册"** + **"工具箱"**

**引用**：强调核心观点
> 上下文窗口是**公共资源**

**列表**：强调多个要点
- 要点 1
- 要点 2
- 要点 3

---

### 4.3 过渡技巧

**章间过渡**：
```markdown
---

## 下一节

从 X 到 Y，我们继续深入...
```

**节间过渡**：
```markdown
理解了 X，接下来看看 Y...
```
